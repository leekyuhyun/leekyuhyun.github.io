# FROM

`FROM`은 베이스 이미지를 생성하는 역할을 한다.
Docker 컨테이너를 특정 초기 이미지 기반으로 추가적인 세팅을 할 수 있다.
여기서 말한 '특정 초기 이미지'가 곧 베이스 이미지이다.

## 사용법

```docker
# 문법
FROM [이미지명]
FROM [이미지명]:[태그명]
// 태그명을 적지 않으면 해당 이미지의 최신(latest) 버전을 사용한다.
```

## 실습 : FROM을 이용하여 노드 이미지 만들기

1. Dockerfile 만들기

```docker
FROM NODE
```

2. Dockerfile 기반 이미지 만드는 명령어

```bash
# docker build -t [이미지명]:[태그명] [Dockerfile이 존재하는 디렉터리 경로]
$ docker build -t my-next-app .
```

3. 이미지를 기반으로 컨테이너 띄우기

```bash
$ docker run -d my-next-app
```

![](https://velog.velcdn.com/images/leekh010502/post/b68b9b9e-6d70-4a5b-83b3-78ad797fb16d/image.png)

4. 컨테이너 조회하기

```bash
$ docker ps # 실행되고 있는 컨테이너가 없다.
$ docker ps -a # 확인해보니 컨테이너가 종료되어 있다.
```

Docker의 컨테이너는 내부적으로 필요한 명령을 다 수행하면 컨테이너가 저절로 종료된다.

5. 컨테이너 내부로 들어가서 node가 잘 깔렸는지 확인해보기
   Dockerfile

```docker
FROM node

ENTRYPOINT ["/bin/bash", "-c", "sleep 500"]
```

```bash
$ docker build -t my-next-app . # 이미지 빌드
$ docker run -d my-next-app # 컨테이너 실행
$ docker ps # 실행 중인 컨테이너 조회
$ docker exec -it [컨테이너 ID] bash # 컨테이너 접속

$ node -version # node 설치되어 있는 지 확인
```

# COPY

## COPY란??

호스트 컴퓨터에 있는 파일을 복사해서 컨테이너로 전달한다.

## 사용법

```docker
# 문법
COPY [호스트 컴퓨터에 있는 복사할 파일의 경로] [컨테이너에서 파일이 위치할 경로]

# 예시
COPY app.txt /app.txt
```

```docker
FROM ubuntu

COPY app.txt /app.txt

ENTRYPOINT ["/bin/bash", "-c", "sleep 500"]
```

# ENTRYPOINT

`ENTRYPOINT`는 컨테이너가 생성되고 최초로 실행할 때 수행되는 명령어를 뜻한다.
쉽게 설명하자면 `ENTRYPOINT`에는 미니 컴퓨터의 전원을 키고나서 실행시키고 싶은 명령어를 적으면 된다.

## 사용법

```docker
# 문법
ENTRYPOINT [명령문...]

# 예시
ENTRYPOINT ["node", "dist/main.js"]
```

## 실습

Dockerfile

```docker
FROM ubuntu

ENTRYPOINT ["/bin/bash", "-c", "echo hello"]
```

```bash
$ docker build -t my-server5 .
$ docker run -d my-server5
$ docker ps -a
$ docker logs [Container ID]
```

# RUN

이미지 생성 과정에서 명령어를 실행시켜야 할 때 사용한다.

## 사용법

```docker
# 문법
RUN [명령문]

# 예시
RUN npm install
```

### ✅ `RUN` vs `ENTRYPOINT`

`RUN` 명령어와 `ENTRYPOINT` 명령어가 헷갈릴 때가 있다.
둘 다 같이 명령어를 실행시키기 때문이다. 하지만 엄연히 둘의 사용 용도는 다르다.
`RUN`은 ‘**이미지 생성 과정**’에서 필요한 명령어를 실행시킬 때 사용하고, `ENTRYPOINT`는 생성된 이미지를 기반으로 **컨테이너를 생성한 직후에** 명령어를 실행시킬 때 사용한다.

## 실습

미니 컴퓨터 환경이 ubuntu로 구성되었으면 좋겠고 git이 깔려있으면 좋겠다고 가정하자. 이런 환경을 구성하기 위해 `Dockerfile`을 활용해 ubuntu, git이 깔려있는 이미지를 만들면 된다.

1. Dockerfile 작성하기
   Dockerfile

```bash
    FROM ubuntu
    RUN apt update && apt install -y git
    ENTRYPOINT ["/bin/bash", "-c", "sleep 500"]
```

2. 이미지 빌드 및 컨테이너 실행

```
$ docker build -t my-server6 .
$ docker run -d my-server6
$ docker exec -it [Container ID] bash

$ git -v # 컨테이너 내에 git이 잘 설치됐는 지 확인
```

# WORkDIR

`WORKDIR`으로 작업 디렉터리를 전환하면 그 이후에 등장하는 모든 `RUN`, `CMD`, `ENTRYPOINT`, `COPY`, `ADD` 명령문은 해당 디렉터리를 기준으로 실행된다.
작업 디렉터리를 굳이 지정해주는 이유는 컨테이너 내부의 폴더를 깔끔하게 관리하기 위해서이다.
컨테이너도 미니 컴퓨터와 같기 때문에 `Dockerfile`을 통해 생성되는 파일들을 특정 폴더에 정리해두는 것이 추후에 관리가 쉽다.
만약 `WORKDIR`을 쓰지 않으면 컨테이너 내부에 존재하는 기존 파일들과 뒤섞여버린다.

## 사용법

```docker
# 문법
WORKDIR [작업 디렉토리로 사용할 절대 경로]

# 예시
WORKDIR /usr/src/app
```

## 실습

1. `app.txt`, `src`, `config.json` 파일 만들기

2. `Dockerfile` 만들어서 이미지 생성 및 컨테이너 실행

   > `WORKDIR`을 안 썼을 때 파일이 어떻게 구성되는지 먼저 확인

   Dockerfile

   ```docker
   FROM ubuntu

   COPY ./ ./

   ENTRYPOINT ["/bin/bash", "-c", "sleep 500"]
   ```


# 웹 프론트엔드 프로젝트 (Next.js)를 Docker로 배포하기

1. Next.js 프로젝트 만들기

```bash
$ npx create-next-app@latest
```

2. Dockerfile 작성하기
   Dockerfile

```docker
FROM node:20-alpine

WORKDIR /app

COPY . .

RUN npm install

RUN npm run build

EXPOSE 3000

ENTRYPOINT [ "npm", "run", "start" ]
```

3. .dockerignore 작성하기
   .dockerignore

```bash
node_modules
```

- 이미지를 생성할 때 npm install을 통해 처음부터 깔끔하게 필요한 의존성만 설치한다.
  따라서 호스트 컴퓨터에 있는 node_modules는 컨테이너로 복사해갈 필요가 없다.

4. Dockerfile을 바탕으로 이미지 빌드하기

```bash
$ docker build -t my-web-server .
```

5. 이미지가 잘 생성됐는지 확인하기

```bash
$ docker image ls
```

6. 생성한 이미지를 컨테이너로 실행시켜보기

```bash
$ docker run -d -p 80:3000 my-web-server
```

7. 컨테이너가 잘 실행되고 있는지 확인하기

```bash
$ docker ps
```
